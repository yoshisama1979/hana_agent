# TDD / BDD 開発インストラクション

## このインストラクションの目的

AIエージェントが以下を守るための **強制ガードレール**。

* 勝手に先に進まない
* テストでズルをしない
* 設計を壊さない

---

## 基本原則（常に守る）

### 確認優先

以下のタイミングでは、**必ずユーザーの承認を得てから**次へ進む。

* BDD シナリオ確定時
* テスト設計確定時
* 各シナリオの Green 到達時
* スコープ拡張や解釈が必要になった場合

### 一歩ずつ進める

* **1シナリオ → 1テスト → 1実装** を厳守する
* 複数シナリオ・複数テストを同時に進めない

### 推測禁止

* 不明点・判断が分かれる点は実装せず、**質問として列挙してユーザーに確認する**

---

## フェーズ1：調査と統合分析

### 目的

実装前に「既存コードにどう接続されるか」を把握する。

### 実施内容

* ディレクトリ構成・命名規則・モジュール分割の確認
* 既存テストの書き方・粒度の確認
* 統合ポイント（外部API・ファイルI/O・DB等）の特定
* 技術的制約・既存パターンの整理

### 成果物（報告内容）

* 変更対象候補
* 依存関係
* 守るべき規約の要点

---

## フェーズ2：対話的 BDD シナリオ作成（承認必須）

### 原則

* **シナリオが承認されるまで、コードを1行も書かない**
* BDD は「人間との合意形成」、TDD は「保証手段」

### シナリオ形式

```
Feature: <機能名>

  Scenario: <シナリオ名>
    Given <前提条件>
    When  <操作・入力>
    Then  <期待される結果>
    And   <追加の期待結果（必要な場合）>
```

### シナリオ要件

* 正常系だけでなく、**異常系・境界条件**も必ず検討する
* 各 Scenario に **受け入れ基準（AC）** を付ける
* 曖昧な表現（「適切に」「なるべく」等）を使わない
* 各 Scenario は **1つ以上のテストに対応**させる
* Scenario に対応しない実装は禁止

→ **ユーザー承認後のみ次フェーズへ進む**

---

## フェーズ3：テスト設計（承認必須）

### 目的

「どのテストで、どの振る舞いを保証するか」を確定する。

### 決定事項

| 項目 | 内容 |
|------|------|
| テスト種類 | Unit / Integration |
| テスト対象 | モジュール・関数・クラス名 |
| 観測点 | 戻り値・例外・ファイル出力・標準出力 等 |
| モック対象 | 外部API・ファイルI/O・DB 等、テストで切り離すもの |

### pytest でのテスト命名規則

```python
# ファイル名: tests/test_<module>.py

def test_<対象>_<条件>_<期待結果>():
    ...
```

例：
```python
def test_parse_csv_with_valid_file_returns_records():
    ...

def test_parse_csv_with_missing_file_raises_file_not_found_error():
    ...
```

→ **ユーザー承認後のみ次フェーズへ進む**

---

## フェーズ4：TDD（RGBC サイクル）

### Red：失敗するテストを書く

* **必ず失敗するテストから始める**
* コンパイルエラーで止まる場合は、クラス・関数を仮定義してよい
* 1テスト = 1振る舞い
* Given / When / Then をコメントで明示する

```python
def test_convert_amount_with_tax_returns_correct_value():
    # Given
    converter = TaxConverter(rate=0.1)

    # When
    result = converter.apply(1000)

    # Then
    assert result == 1100
```

**チェック：**
* `uv run pytest` を実行し、テストが **意図した理由で失敗**しているか確認する
* 「関数が存在しない」以外の理由で落ちている場合は原因を調査する

---

### Green：最小限の実装で通す

* 目的は **テストを通すことだけ**
* 過剰設計・最適化・先読み実装は禁止

**ズル禁止（以下は不可）：**

```python
# NG: テスト値に依存したハードコード
def apply(self, amount):
    return 1100  # テストの期待値をそのまま返している

# NG: 常に固定値を返す実装
def apply(self, amount):
    return amount  # taxを一切考慮していない

# NG: 条件分岐の丸め込み
def apply(self, amount):
    if amount == 1000:
        return 1100
```

**テストコードの変更について：**

原則、テストコードは変更しない。以下の場合のみ例外とする。

* BDD シナリオの解釈ミスが判明した
* 受け入れ基準との矛盾が生じた
* 実装詳細に依存しすぎているテストだった

→ 必ず **理由を説明 → ユーザー承認 → 修正** の順で行う

→ **Green 到達ごとにユーザーに確認を取る**

---

### Blue：リファクタリング（品質向上）

テストがグリーンのまま、以下を確認・実施する。

* 重複の排除（DRY）
* 命名の明確化
* ネストの削減（ガード節への書き換え）
* 責務の分離（1関数1責務）
* マジックナンバーの定数化
* 型ヒントの付与
* docstring の整備（必要な場合）

**挙動変更は禁止。** リファクタ後に `uv run pytest` を実行し、全テスト通過を確認する。

---

### Commit：小さく、意味のある単位で

* **テストがグリーンの状態のみ**コミットする
* 1コミット = 1論理変更
* メッセージ形式：`<type>: <何を・なぜ>`

```
# 例
feat: CSVパース関数の正常系を実装
test: TaxConverter の境界値テストを追加
refactor: apply() の条件分岐をガード節に整理
```

---

## フェーズ終了時の報告フォーマット（必須）

各フェーズ終了時に、以下の形式で必ず報告する。

```
## フェーズ報告

- 現在のフェーズ  ：フェーズN（○○）
- 実施内容の要約  ：（箇条書きで簡潔に）
- テスト結果      ：全X件 PASSED / X件 FAILED（FAILEDの場合は内容を記載）
- 次のアクション  ：（次フェーズで行うこと）
- 続行確認        ：次のフェーズへ進んでよいですか？
```

---

## よくある失敗の防止ルール

| 状況 | 対処 |
|------|------|
| テストが大きすぎる | Given/When/Then の単位で分割する |
| Red → Green に進めない | 仮実装（ハードコード）から始め、徐々に汎化する |
| リファクタ中にテストが落ちた | 即ロールバックして原因を調査する |
| 外部依存のテストが不安定 | `unittest.mock.patch` で切り離す |
| パフォーマンス最適化したい | 全テスト通過後、最後に行う |

---

## 役割分担（変えない）

* **人間**が「判断」する
* **AI** が「実装」する
* **テスト**が「保証」する

この分担を崩さないこと。
