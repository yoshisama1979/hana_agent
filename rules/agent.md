# エージェント向けルール

## 必ず最初に読むこと

- **このファイル（rules/agent.md）を最初に読み込んでから作業を開始すること。**
- プロジェクト実装時は、**rules ディレクトリ内の各ルールファイルを確認**したうえで作業すること。

## 仕様書の場所

- 本プロジェクトの仕様書は **documents/document.md** に記載されている。
- 機能要件・処理仕様・データ仕様などを確認する際は、必ず documents/document.md を参照すること。

## 一つの機能の実装フロー（進め方）

**重要**: 一つ一つのステップが完了するごとに、必ずユーザーにフェーズを報告し、次のステップへ進むか確認すること。

| 順 | ステップ | 実施内容 | 完了時のアクション |
|----|----------|----------|---------------------|
| 1 | **仕様の記載** | ユーザーが「何を作りたいか」を伝える → その内容をもとに **documents/document.md（仕様書）** に仕様を書き込む。 | 仕様記載完了を報告し、**ユーザーにフェーズを移す**。 |
| 2 | **実装（TDD）** | 記載した仕様を基に、**TDD** で実装する。rules/tdd.md および rules/testcode.md に従う。 | 実装完了を報告し、**ユーザーにフェーズを移す**。 |
| 3 | **コミット準備** | ユーザーがファイルをステージングする。**ステージングされたファイルの差分**を確認し、問題がなければ**コミットメッセージ**を提案する。 | 提案完了を報告し、**ユーザーにフェーズを移す**。 |
| 4 | **リファクタリング** | **rules/refactoring.md** を確認し、それに従って**ステージングされたファイル**に対してリファクタリングを行う。 | リファクタリング完了を報告し、**ユーザーにフェーズを移す**。 |
| 5 | **動作確認・仕様書更新** | ユーザーが動作を確認する。その後、**仕様書（documents/document.md）を再度更新**する。 | 更新完了を報告し、**ユーザーにフェーズを移す**。 |

以上で**一連の一つの機能の実装は完了**とする。次の機能に進む場合はステップ1から繰り返す。

> ※ フロントエンドが存在しないため、元の「フロントエンド実装」「フロントエンドのコミット準備・リファクタリング」ステップは省略している。

## 機能実装時にAIがすべきこと（共通原則）

フローに加え、以下を守ること。

### 実装前

- **仕様の確認**: documents/document.md およびユーザーの指示を読み、実装範囲と受け入れ条件を把握する。
- **曖昧さの解消**: 仕様や要件が不明な点があれば、**実装せずに質問として列挙**し、ユーザーの回答を得てから進める。
- **既存コードの把握**: 既存のディレクトリ構成・命名規則・コーディング規約を確認し、それに合わせて実装する。

### 実装中

- **仕様の範囲内で実装**: 仕様書に書かれていない機能や仕様の拡張は、ユーザーが明示的に依頼するまで行わない。
- **推測禁止**: 判断が分かれる点・解釈が複数ある点は、推測で実装せず質問する（rules/tdd.md に準拠）。
- **テストの実行確認**: 実装後、`uv run pytest` を実行し、**全テストが成功していることを確認**してからフェーズを移す。
- **型ヒントの付与**: 関数・メソッドには原則として型ヒント（Type Hints）を付ける。
- **エラー・境界の考慮**: 仕様に異常系・バリデーション・境界値が含まれる場合は、それらを実装とテストに反映する。

### Python固有のルール

- **モジュール構成**: 機能ごとにモジュールを分割し、`src/` レイアウトを基本とする。
- **外部依存の切り離し**: ファイルI/O・外部API・DB等は `mock` / `patch` でテスト時に切り離す。
- **設定管理**: APIキー・パスワード等は `.env` ファイルで管理し、`python-dotenv` 等で読み込む。ソースコードへのハードコード禁止。
- **Ruffによる整形**: コミット前に `uv run ruff check` および `uv run ruff format` を実行し、警告がないことを確認する。
- **依存関係の追加**: 新規パッケージを追加する場合は `uv add <package>` を使い、理由と用途を簡潔に説明する。

### セキュリティ・環境

- **秘密情報の扱い**: APIキー・パスワード・トークン等をソースコードにハードコードしない。`.env` で管理し、`.env.example` に項目のみ記載する。
- **`.gitignore`の確認**: `.env` および `__pycache__/`、`.venv/` 等が `.gitignore` に含まれていることを確認する。

### コミット準備時（差分確認）

- **問題がある場合**: ステージングされた差分に、明らかなバグ・仕様違反・セキュリティリスク・不要な変更が含まれる場合は、**コミットメッセージを提案せず**、問題点を指摘してユーザーに報告する。
- **問題がない場合**: 差分が仕様に沿っており問題ないと判断したときのみ、コミットメッセージを提案する。
