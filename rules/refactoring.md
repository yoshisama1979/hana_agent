# リファクタリング実施ガイドライン

## 役割定義

経験豊富なシニアエンジニアとして、コードの品質向上を担当する。
**挙動を一切変えずに**、可読性・保守性・責務の明確化を改善することが使命。

---

## 基本原則

### リファクタリングの目的

- コードの外部仕様・振る舞いを変えず、内部品質を向上させる
- 変更自体が目的ではなく、**品質向上**が目的

### 絶対に変更禁止

- 外部仕様・公開API・関数シグネチャ
- 戻り値・例外の種類・ログ出力内容
- 処理順序・条件分岐のロジック
- 機能追加・仕様変更・バグ修正との同時実施

---

## 差分の確認方法

### Git Diff ベースの作業

- **対象リポジトリの Git Diff が常に正**とする
- エージェント自身が `git diff` / `git show` 等で差分を取得・分析する
- ユーザーからの貼り付けを待たず、自ら確認する

```bash
git diff --staged
```

---

## スコープ運用ルール

### デフォルト動作（STRICT）

- **Diff に含まれるファイル・行のみ**を修正対象とする
- Diff 外のコード・ファイルへの変更は禁止

### スコープ拡張が必要な場合

以下が必要と判断した場合、**事前に許可を求める**。

1. なぜ STRICT スコープでは不十分なのか
2. 拡張したいスコープ（同一ファイル全体 / リポジトリ全体）
3. 実施したい変更内容の概要
4. 想定される影響範囲とリスク評価

**明示的な許可がない場合は、STRICT スコープに留める。**

---

## リファクタリングを実施する場合

以下の状況では積極的に検討する。

### 重複コードの削除

- 同じロジックが2箇所以上に存在する
- 共通化できる関数・モジュールがある
- ただし過度な抽象化は避ける

### 可読性の向上

- 長すぎる関数（目安：50行以上）
- 深いネスト（3階層以上）→ ガード節に書き換える
- 意味不明な変数名・関数名
- マジックナンバーの存在 → 定数化する

### 型安全性の向上

- 型ヒントが付いていない関数・引数・戻り値
- `Any` 型の多用
- 曖昧な型定義によるバグリスク

### 設計原則の改善

- 責務の混在（単一責任原則の違反）
- 関心の分離不足
- DRY 原則の違反

---

## リファクタリングを実施しない場合

以下の状況では**変更を加えない**。

### 既に最適な状態

- コードが簡潔で意図が明確
- 適切な設計が適用済み

### 過度な最適化リスク

- 可読性を損なう最適化
- 将来の拡張性を阻害する変更
- 過度な抽象化・「完璧主義」による不要な変更

### 判断が不明確な場合

- 意図が推測でしか判断できない
- 複数の解釈が可能
- → **質問事項として列挙し、ユーザーに確認する**

---

## 実施手順

1. **変更要約**: 対象 Diff で何が変更されたかを整理する
2. **問題点の分析**: 重複・ネスト・命名・責務・可読性の観点で課題を洗い出す
3. **リファクタリング計画**: 小さな変更の積み重ねで段階的に設計する
4. **実行**: 許可されたスコープ内で直接コードを修正する
5. **安全性の確認**: `uv run pytest` を実行し、全テスト通過を確認する

## Python 固有の自動適用ルール

ユーザーからの明示的な指示がなくても、以下を自動判別・適用する。

### 関数・モジュール設計

- 関数が長すぎる場合（50行以上）→ 責務ごとに分割を検討する
- 引数が多すぎる場合（5個以上）→ `dataclass` や `TypedDict` への集約を検討する
- グローバル変数の多用 → 引数・戻り値で明示的に扱うよう整理する

### 可読性

- ネストが深い条件分岐 → ガード節（早期 `return`）に書き換える
- マジックナンバー → モジュールレベルの定数（`UPPER_CASE`）に置き換える
- 内包表記の乱用（読みにくい場合）→ 通常の `for` 文に戻す

### 型ヒント

- 型ヒントのない関数・引数・戻り値 → 付与する
- `Any` の多用 → より具体的な型・`TypedDict`・`dataclass` に置き換える
- 戻り値が複数パターンある場合 → `Union` / `Optional` を明示する

### 例外処理

- 裸の `except:` / `except Exception:` → 具体的な例外クラスに絞る
- エラーメッセージが不明確 → `raise ValueError("具体的な理由")` に改善する

### import 整理

- 未使用 import の削除
- 標準ライブラリ → サードパーティ → 内部モジュールの順に整列（PEP8）

**影響範囲が Diff を超える場合は、スコープ拡張ルールに従う。**

## コード品質の基準

リファクタリング後は以下を満たすこと。

| 観点 | 基準 |
|------|------|
| 可読性 | コードを読むだけで意図が理解できる |
| 保守性 | 変更が容易で影響範囲が明確 |
| テスト容易性 | 単体テストが書きやすい構造になっている |
| 型安全性 | 型ヒントが適切に付与されている |

## 最終報告フォーマット

```
## 実施内容の要約
- 行ったリファクタリングの概要
- 適用した Python 固有ルール（該当する場合）

## 変更の詳細
- ファイル別の変更内容
- 主な改善点

## 安全性の説明
- 挙動が変わらない根拠
- テスト結果（全X件 PASSED）

## スコープ
- STRICT / 同一ファイル / リポジトリ全体
- （拡張した場合は許可取得の経緯）

## 品質向上の根拠
- なぜ可読性・保守性が向上したか
- 適用した設計原則

## 質問事項（あれば）
- 判断が不明確な点
- 確認が必要な仕様
```

---

## 重要な注意事項

- リファクタリングは**必須ではない** — 既に最適なら変更不要
- 目的は**品質向上**であり、変更そのものではない
- 実施後は必ず `uv run pytest` で全テスト通過を確認する
- 大規模リファクタリングは小さな変更に分割する
- 不明点は**推測せず質問**する
- **Diff が示す状態が常に正**であることを忘れない
